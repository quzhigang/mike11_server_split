# 项目介绍与源码拆分计划

## 1. 文档目的

本文档用于说明当前项目的背景、主要业务职责、超大源码文件治理目标，以及适合交给 LLM 分阶段执行的拆分步骤。

本轮治理的首要对象是 `Mike11/Res11.cs`。目标是在 **不改变现有业务行为、数据库交互行为、模型文件读写行为和对外接口** 的前提下，按业务功能模块完成拆分，提升代码的可读性、可维护性和调试效率。

## 2. 项目简介

本项目是一个基于 `ASP.NET + .NET Framework 4.6.1` 开发的水利专业模型后台服务，用于为 B/S 架构的数字孪生系统提供模型计算与结果处理能力。

项目当前主要依赖 DHI 的 MIKE 系列模型引擎，重点包括：

- `MIKE 11`
- `MIKE 21`
- `MIKE FLOOD`

系统主要承担以下业务能力：

- 方案管理
- 模型参数设置
- 模型输入文件读写
- 模型计算引擎调用
- 模型结果提取
- 模型结果统计分析
- GIS 化成果输出

## 3. 当前存在的问题

当前项目中存在多个体量较大的核心源码文件，例如：

- `Mike11/Res11.cs`
- `Mike11/Nwk11.cs`
- `Mike11/Reach.cs`
- `HydroModel.cs`
- `Model_Ser.ashx.cs`

以 `Res11` 为例，典型问题包括：

- 单文件职责过多
- 方法分布过散
- 主方法与业务子方法相距较远
- 调试时来回跳转
- 新增逻辑难以判断应放入哪个位置
- 文件越来越大后，继续重构的风险不断上升

## 4. 本轮治理目标

本轮源码拆分治理的目标如下：

1. 将大文件按业务主题拆分为多个职责清晰的源码文件
2. 保持类名、命名空间、对外方法签名尽量不变
3. 保持现有业务功能行为不变
4. 建立后续可以重复套用的拆分步骤，用于治理 `Nwk11.cs`、`Reach.cs`、`HydroModel.cs`

## 5. 总体拆分原则

这些原则是总要求，但不建议一次性全部压给 LLM 同时执行。更适合按阶段拆开，每一步只强调其中几条。

### 5.1 文件大小控制

- 一个源码文件尽量不超过 `2000` 行
- 较合适区间为 `1000-2000` 行
- 不为了平均行数而机械拆分

### 5.2 按业务主题拆分

- 拆分的首要依据是业务功能模块，不是平均分配代码行数
- 业务逻辑紧密相关的代码不要强行拆散
- 一个源码文件尽量只承载一个主要业务主题

### 5.3 类尽量保持完整

- 一个类尽量保持完整
- 过大时优先使用 `partial class`
- `partial` 文件命名应直接体现业务职责

### 5.4 主文件不应为空壳

- 主文件应保留真正核心的业务入口
- 不建议保留只有几行定义的空壳主文件
- 主文件应能代表该类的主要阅读入口

### 5.5 主文件应尽量瘦身

- 主文件不应继续堆积大量辅助逻辑
- 时间序列辅助、累计量、格式转换、只服务于某个业务块的辅助方法，应尽量拆出去

### 5.6 调试顺手优先

- 一个方法调用的业务子方法，只要不是纯公共通用方法，尽量放在同一个源码文件
- 同一源码文件中，主方法后面尽量紧跟它的业务子方法
- 子方法之间尽量按调用顺序排列

### 5.7 公共方法不要过度抽离

- 真正和业务主题无强关联、可被多个模块复用的通用方法，才适合抽离
- 如果一个辅助方法虽然短小，但明显只服务于某个业务主流程，则更适合跟着该业务文件放置

### 5.8 拆分文件多时要建立子目录

- 如果一个类拆分出的文件较多，应建立专用目录，例如 `Mike11/Res11/`
- 避免一级目录下散落过多同类文件

## 6. 为什么不适合一次性把全部规则都压给 LLM

如果把所有规则一次性压给 LLM，容易出现以下问题：

- 同时满足“业务边界”“文件大小”“主文件瘦身”“方法顺序”“调试顺手”时，约束会互相冲突
- 模型容易为了满足某一条规则而破坏另一条规则
- 一次改动范围过大，人工复核和静态检查成本迅速上升
- 很容易出现“拆得很碎，但调试更不方便”的反效果

因此更合理的方式是：

**按阶段拆分，每一阶段只强调少量关键规则，完成一轮后再进入下一轮。**

## 7. 适合 LLM 执行的拆分步骤

以下步骤适合一轮一轮执行。每一步只要求 LLM 遵守当前步骤的少量规则，完成后再进入下一步。

### 第 1 步：识别业务模块，不改代码

目标：

- 通读目标大文件
- 列出主要业务块
- 给方法按业务主题分组
- 标出哪些方法明显应放在一起

本步只要求遵守的规则：

- 按业务功能模块识别，不按行数拆
- 不修改代码逻辑
- 不移动方法
- 先输出分组和拆分草案

本步输出物：

- 方法清单
- 业务模块分组表
- 初步拆分草案

### 第 2 步：先拆低风险模块，建立 partial 结构

目标：

- 创建拆分目录和基础 `partial class` 结构
- 优先迁移低风险、低耦合模块

建议优先处理：

- 模型与枚举定义
- 序列化与反序列化
- 通用转换
- 时间序列辅助

本步只要求遵守的规则：

- 类名、命名空间、对外接口保持不变
- 优先拆低耦合模块
- 不碰核心主流程
- 文件命名体现业务职责

本步验收标准：

- 拆分目录与 `partial` 结构已建立
- 低耦合模块已从主文件中分离
- 原类对外访问方式保持一致

### 第 3 步：拆独立业务模块

目标：

- 将业务边界较清晰的模块拆出去

建议优先处理：

- GIS
- 风险分析
- 预警与巡查
- 闸门状态与闸门流量
- 统计报表
- 结果查询

本步只要求遵守的规则：

- 主方法和直接业务子方法尽量在同一文件
- 一个文件只保留一个主要业务主题
- 单文件尽量控制在 `2000` 行以内
- 如现有文件不合适，可以新增更贴业务的新文件

本步验收标准：

- 拆出的文件业务主题明确
- 主文件体积明显下降
- 调试某一业务流程时，不必跨过多文件跳转

### 第 4 步：主文件瘦身

目标：

- 让主文件只保留核心入口和最常阅读的主流程

适合继续移出的内容：

- 时间序列处理
- 累计量、体积、格式转换
- 明显只服务于某个业务块的辅助方法

本步只要求遵守的规则：

- 主文件不保留无关辅助逻辑
- 主文件保留核心主流程，不做空壳
- 能明显归属其他业务主题的方法，尽量移到对应文件

本步验收标准：

- 主文件显著比初始状态更轻
- 打开主文件能直接看到核心业务入口

### 第 5 步：按“调试顺手”标准重新归并

目标：

- 重新检查拆分后的各文件
- 重点不是继续拆，而是修正“方法放错文件”的问题

本步重点检查：

- 某个方法是否应该放到更合适的业务文件
- 主方法调用的业务子方法是否分散在多个文件里
- 是否存在过度抽离所谓“公共方法”的情况

本步只要求遵守的规则：

- 业务子方法只要不是纯公共通用方法，尽量和主方法在同一文件
- 如果某文件业务主题混杂，应重新归类
- 允许新建更小、更贴业务的文件

本步验收标准：

- 每个文件的业务主题更纯
- 调试一个业务流程时，跨文件跳转明显减少

### 第 6 步：只做版面整理

目标：

- 在业务归类基本稳定后，再做版面优化
- 调整各文件内部的方法顺序

本步只要求遵守的规则：

- 主方法后面尽量紧跟它调用的子方法
- 子方法之间尽量按调用顺序排列
- 不改业务逻辑，只调整方法位置

本步验收标准：

- 从上往下阅读文件时，可以顺着调用链读
- 调试时不用前后翻找方法定义

### 第 7 步：统一检查和验收

目标：

- 进行最终结构复核
- 检查是否产生新的“杂项大文件”

本步重点检查：

- 单文件是否仍明显过大
- 是否还有业务主题不清晰的文件
- 是否有应归并却被拆散的逻辑
- 是否仍有应拆出却混在主文件中的逻辑

本步只要求遵守的规则：

- 关注整体结构质量
- 不为追求形式而过度重构
- 以可读性、可维护性、调试便利性为最终标准

## 8. 推荐给 LLM 的任务下发方式

建议每次只给 LLM 一个明确目标，而不是一次性要求“全部拆完”。

### 示例 1：先做业务分组

> 先不要改代码，通读 `Res11.cs`，按业务功能分组所有方法，给出拆分草案。

### 示例 2：只拆低风险模块

> 根据前面的分组，先拆出模型定义、序列化、格式转换、时间序列辅助，不要动核心结果读取流程。

### 示例 3：只做主文件瘦身

> 继续减小 `Res11.cs`，把明显属于其他业务模块的辅助方法移走，但主文件保留核心入口。

### 示例 4：只做调试友好整理

> 重新检查已拆分文件，把主方法和它的业务子方法尽量放回同一个文件，不要做新的业务重构。

### 示例 5：只做版面修正

> 不改逻辑，只调整方法顺序，确保各子方法按调用顺序放在主方法后面。

## 10. 验收标准

### 10.1 结构验收

- 大文件已拆分为多个职责清晰的源码文件
- 单文件原则上不超过 `2000` 行
- 文件命名与业务职责一致
- 目录结构清晰

### 10.2 功能验收

- 计算结果不变
- 数据库存取行为不变
- 文件读写行为不变
- 对外调用方式基本不变

### 10.3 可维护性验收

- 主文件不再堆积全部逻辑
- 各模块业务边界明确
- 主方法和业务子方法更集中
- 调试跨文件跳转明显减少

## 11. 后续建议

建议将 `Res11` 的拆分作为标准样板，后续按同样步骤继续治理：

- `Mike11/Nwk11.cs`
- `Mike11/Reach.cs`
- `HydroModel.cs`
- `Model_Ser.ashx.cs`

并逐步形成项目级治理约束：

- 超过 `2000` 行的核心业务文件进入治理清单
- 新增代码优先落入职责明确的文件
- 大型业务类优先采用“专用目录 + partial”方式组织
