# 项目介绍与源码拆分计划

## 1. 目标

在不改变现有业务行为、数据库交互、模型文件读写流程和对外接口的前提下，持续治理大文件，提升可读性、可维护性和调试效率。

当前重点文件：

- `Mike11/Res11.cs`
- `Mike11/Nwk11.cs`
- `Mike11/Reach.cs`
- `HydroModel.cs`
- `Model_Ser.ashx.cs`

## 2. 本项目拆分口径

- 优先按业务主题拆分
- 若原文件已有明确 `#region / #endregion`，第一轮优先按这些边界做粗拆
- 主文件保留核心业务入口，不做空壳
- 与主流程强相关的辅助方法可以保留或并回主文件，但应放在主文件末尾独立 `region`
- 不为压缩行数而强行拆散主方法和直接子方法
- 一个文件尽量只承载一个主要业务主题
- 是否需要修复“乱码注释”，以 IDE 或文本编辑器中的实际显示为准

## 3. 执行步骤

### 第 1 步：识别模块

- 通读目标文件
- 按业务主题分组方法
- 同时标出原始 `region` 边界
- 输出拆分草案

### 第 2 步：第一轮粗拆

- 按原始 `region` 先做粗拆
- 建立子目录和 `partial class` 结构

### 第 3 步：低风险拆分

- 拆模型、枚举、列表类、序列化、转换等低风险部分

### 第 4 步：独立业务块拆分

- 按业务主题拆 GIS、闸门、统计、查询、风险等模块
- 主方法和直接子方法尽量保持在同一文件

### 第 5 步：主文件瘦身与归并

- 将明显不属于主流程的部分移出
- 若拆分过散，允许把强相关辅助方法并回主文件末尾独立 `region`

### 第 6 步：只做版面整理

- 调整方法顺序
- 让主方法和子方法按调用链排列
- 不改逻辑

## 4. 验收标准

### 结构验收

- `partial class` 声明一致
- 工程文件编译项完整
- 无重复方法定义
- 无命名空间冲突
- 主文件仍保留核心业务入口
- 业务分片职责边界清晰

### 验证方式

当前拆分阶段默认只做静态复核，不做编译验证。

静态复核至少包括：

- 工程文件引用检查
- 重复定义检查
- 命名冲突检查
- 主文件职责检查

## 5. 推荐输出形式

每次拆分任务建议只完成一个明确阶段，并输出：

- 当前拆分范围
- 新增或调整的分片文件
- 主文件保留了哪些入口
- 本轮静态复核结果
