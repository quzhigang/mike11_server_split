# AGENTS.md

本文件用于约束当前项目中的代码修改方式，尤其是大文件拆分。

## 1. 基本要求

- 所有输出与说明均使用中文
- 优先保持现有业务行为不变
- 优先保持类名、命名空间、对外方法签名不变
- 不要在一次修改中同时做“重构 + 逻辑改造 + 风格清洗”

## 2. 当前治理对象

当前重点治理以下大文件：

- `Mike11/Res11.cs`
- `Mike11/Nwk11.cs`
- `Mike11/ItemInfo.cs`
- `HydroModel.cs`
- `Model_Ser.ashx.cs`

## 3. 拆分总规则

- 单文件尽量不超过 `2000` 行，合适区间为 `1000-2000` 行
- 优先按业务模块拆分，不按平均行数拆分
- 若原文件已有明确 `#region / #endregion` 边界，第一轮优先按这些边界做粗拆
- 业务逻辑紧密的代码不要强行拆散
- 一个类尽量保持完整，过大时使用 `partial class`
- 主文件应保留核心业务入口，不应为空壳
- 与主流程强相关的辅助方法允许保留或重新并回主文件，但应放在主文件末尾独立 `region`
- 纯公共、纯通用方法才适合单独抽离
- 一个方法调用的业务子方法，只要不是纯公共通用方法，尽量放在同一源码文件
- 文件内部尽量保持“主方法在前，业务子方法按调用顺序在后”

## 4. 推荐步骤

### 第 1 步：识别模块

- 通读目标文件
- 按业务主题分组方法
- 同时识别原始 `region` 边界
- 输出拆分草案

### 第 2 步：先做粗拆和低风险拆分

- 先按原始 `region` 做第一轮粗拆
- 再拆模型、枚举、列表类、序列化、转换等低风险部分
- 建立 `partial class` 结构和子目录

### 第 3 步：拆独立业务块

- 按业务主题拆到专用分片
- 主方法和直接子方法尽量留在同一文件

### 第 4 步：主文件瘦身

- 主文件保留核心入口
- 仅将明显属于其他业务块的辅助方法移出

### 第 5 步：按调试便利性重新归并

- 检查是否拆得过散
- 必要时把强相关辅助方法并回主文件或对应业务文件

### 第 6 步：只做版面整理

- 调整方法顺序
- 让主方法后面紧跟其子方法
- 不改业务逻辑，不改接口

## 5. 修改约束

### 5.1 不轻易改动的内容

- 数据库字段名与表交互逻辑
- 外部调用入口
- DHI MIKE 相关结果读取行为
- 现有模型文件读写流程

### 5.2 允许优先整理的内容

- 方法归类
- `partial` 拆分
- 文件命名
- 目录结构
- 文件内方法顺序

## 6. 收尾复核

拆分收尾阶段默认只做静态复核，不做编译验证。复核重点包括：

- 是否出现重复方法定义
- 是否出现命名空间不一致
- 是否出现 `partial class` 声明不一致
- 是否把业务方法错误归入公共工具文件
- 是否导致主文件再次膨胀
- 工程文件编译项是否完整

## 7. 调试友好要求

- 同一业务流程尽量集中在一个文件
- 主方法和业务子方法尽量不要跨多个文件散落
- 文件内从上往下阅读时，应尽量顺着调用链读下去
- 是否需要修复“乱码注释”，以 IDE 或文本编辑器中的实际显示为准，不以终端显示为准
