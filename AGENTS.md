# AGENTS.md

本文件用于说明在当前项目中开展代码修改，尤其是大文件拆分时，agent 应遵守的项目级规则。

## 1. 基本要求

- 任何输出和说明均使用中文
- 优先保持现有业务行为不变
- 优先保持类名、命名空间、对外方法签名不变
- 不要在一次修改中同时做“重构 + 逻辑改造 + 风格清洗”

## 2. 项目背景

本项目是基于 `ASP.NET + .NET Framework 4.6.1` 的水利专业模型后台服务，主要面向：

- `MIKE 11`
- `MIKE 21`
- `MIKE FLOOD`

核心职责包括：

- 模型计算引擎调用
- 输入文件读写
- 结果提取
- 统计分析
- GIS 化输出

## 3. 当前代码治理重点

当前重点是治理以下大文件：

- `Mike11/Res11.cs`
- `Mike11/Nwk11.cs`
- `Mike11/Reach.cs`
- `HydroModel.cs`
- `Model_Ser.ashx.cs`

## 4. 大文件拆分总规则

这些规则是总原则，但执行时不要一次全部硬压给模型。

- 单文件尽量不超过 `2000` 行，合适区间为 `1000-2000` 行
- 首先按业务模块拆分，不按平均行数拆分
- 业务逻辑紧密的代码不要强行拆散
- 一个类尽量保持完整，过大时使用 `partial class`
- 主文件应保留核心业务入口，不应为空壳文件
- 纯公共、纯通用方法才适合单独抽离
- 一个方法调用的业务子方法，只要不是纯公共通用方法，尽量放在同一源码文件
- 各文件内部的方法顺序，应尽量符合“主方法在前，业务子方法按调用顺序在后”

## 5. 推荐拆分步骤

### 第 1 步：识别业务模块

目标：

- 通读目标大文件
- 给方法按业务主题分组
- 输出拆分草案

本步不要做：

- 不移动代码
- 不修改逻辑
- 不新建文件

### 第 2 步：先拆低风险模块

优先拆：

- 模型与枚举定义
- 序列化与反序列化
- 通用转换
- 时间序列辅助

本步重点：

- 建立 `partial class` 结构
- 文件命名体现业务职责
- 不碰核心读取主流程

### 第 3 步：拆独立业务模块

优先考虑：

- GIS
- 风险分析
- 预警与巡查
- 闸门流量与状态
- 统计报表

本步重点：

- 主方法和直接业务子方法尽量放在同一文件
- 一个文件只保留一个主要业务主题

### 第 4 步：主文件瘦身

本步目标：

- 让主文件只保留核心入口和最常阅读的主流程

适合移出：

- 时间序列辅助
- 累计量、体积、格式转换
- 仅服务于某个业务块的辅助方法

### 第 5 步：按调试便利性重新归并

本步重点：

- 检查是否有方法放错文件
- 检查主方法和业务子方法是否被拆得过散
- 检查是否过度抽离了所谓“公共方法”

### 第 6 步：只做版面整理

本步只允许：

- 调整方法顺序
- 让主方法后面紧跟其子方法
- 让子方法顺序尽量符合调用链

本步不要做：

- 不改业务逻辑
- 不改接口

## 6. 针对当前项目的修改约束

### 6.1 不能轻易改的内容

- 数据库字段名与表交互逻辑
- 外部调用入口
- DHI MIKE 相关结果读取行为
- 现有模型文件读写流程

### 6.2 允许优先整理的内容

- 方法归类
- `partial` 拆分
- 文件命名
- 目录结构
- 文件内方法顺序

### 6.3 修改后应检查

- 是否出现重复方法定义
- 是否出现命名空间不一致
- 是否出现 `partial class` 声明不一致
- 是否把业务方法错误归入公共工具文件
- 是否导致主文件再次膨胀

## 7. 调试友好要求

项目当前很强调“看代码顺手、调试顺手”，因此要优先满足：

- 同一业务流程尽量集中在一个文件
- 主方法和业务子方法尽量不要跨多个文件散落
- 文件内从上往下阅读时，应尽量顺着调用链读下去
